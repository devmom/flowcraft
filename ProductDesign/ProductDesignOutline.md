**一、总体定位要求**

1. 框架必须完全基于 Harness 设计理念，而不是传统 Agent 框架理念。
2. Harness 的核心目标是：可控、可审计、可观测、可编排、可恢复、可治理、可扩展。
3. 从 0 到 1 自研核心框架，不使用 LangChain、AutoGen、CrewAI、Semantic Kernel、Dify 等已有 Agent 框架作为底座。
4. 可以使用通用基础库，例如 HTTP 客户端、数据库、向量数据库、模型 SDK、GUI 框架、安装打包工具，但这些库不能反向决定 Agent 架构。
5. 框架定位是个人和小团队的“本地 AI 工作流助手平台”，不是只面向程序员的开发框架。
6. 默认用户没有编程经验，因此交互、安装、配置、报错、权限确认都必须尽量图形化、向导化、低门槛。
7. 项目以开源形式发布，但要具备产品化体验，不能只是开发者玩具。

---

**二、核心架构要求**

框架至少要包含以下核心层：

```text
Application Layer
面向用户的桌面端、Web UI、CLI、API。

Harness Control Plane
意图识别、任务路由、策略治理、权限控制、人工确认。

Agent Runtime
任务状态机、执行器、调度器、步骤管理、失败恢复。

Capability Gateway
模型网关、工具网关、文件系统网关、浏览器网关、应用控制网关。

Memory System
会话记忆、任务记忆、长期记忆、知识库记忆。

Observability & Audit
日志、追踪、回放、成本统计、工具调用审计。

Infrastructure
本地数据库、配置存储、插件系统、任务队列、沙箱环境。
```

要求 AI 明确说明每层职责、边界、输入输出和模块关系。

---

**三、Harness 设计原则要求**

技术方案必须贯彻以下原则：

1. Agent 不直接调用工具，必须经过 Tool Harness。
2. Agent 不直接调用模型，必须经过 Model Gateway。
3. Agent 不直接读写长期记忆，必须经过 Memory Policy。
4. Agent 不直接执行高风险本地操作，必须经过权限和人工确认。
5. 所有任务必须有 Task State，可以暂停、恢复、失败重试和回放。
6. 所有工具调用必须记录原因、输入、输出、风险等级和执行结果。
7. 所有长期记忆写入必须可解释、可查看、可删除。
8. 用户必须能看到 Agent 正在做什么、为什么做、是否有风险。
9. 默认最小权限，按任务临时授权。
10. 失败是框架的一等公民，必须有失败分类、降级和恢复策略。

**四、本地电脑操控要求**

框架主要运行在 Windows，兼容 Linux。

Windows 优先支持：

```text
文件读写
文件搜索
应用启动
浏览器控制
剪贴板操作
截图识别
Office 文档处理
PDF 处理
命令执行
本地脚本执行
系统通知
计划任务
```

Linux 兼容支持：

```text
文件系统
Shell
浏览器自动化
本地服务
脚本任务
通知能力
```

必须要求：

1. 本地操作全部经过 Tool Harness。
2. 修改、删除、移动文件前必须展示影响范围。
3. 执行命令前必须解释命令用途和风险。
4. 支持沙箱目录，默认只允许在授权目录内操作。
5. 支持白名单和黑名单目录。
6. 支持操作回滚设计，例如文件备份、任务快照、撤销记录。
7. 高风险动作需要用户确认，例如删除文件、覆盖文件、发送邮件、执行脚本、安装软件、访问敏感目录。

**五、用户体验要求**

目标用户是个人、小团队、中小企业，很多人不懂编程。

因此方案必须包含：

1. Windows 一键安装包。
2. 首次启动引导。
3. 图形化配置模型 API Key。
4. 图形化选择本地工作目录。
5. 图形化授权工具权限。
6. 图形化查看任务执行过程。
7. 图形化查看记忆和删除记忆。
8. 图形化插件管理。
9. 图形化日志和错误提示。
10. 小白能理解的风险确认提示。

安装体验要求：

```text
下载安装包
双击安装
启动应用
选择模型供应商
填写 API Key 或选择本地模型
选择工作目录
完成初始化
开始使用
```

不能要求普通用户手动创建虚拟环境、安装依赖、配置环境变量或运行复杂命令。

至少设计支持：

```text

Azure OpenAI
Anthropic
Gemini

```

Model Gateway 必须支持：

1. 模型路由。
2. 成本统计。
3. Token 预算。
4. 失败重试。
5. 模型降级。
6. 结构化输出校验。


必须识别：



要求支持澄清机制：

当用户目标模糊、对象缺失、风险过高或约束冲突时，Agent 必须先向用户提问，而不是直接执行。

**九、任务规划要求**

Planner 必须生成可执行计划，而不是文字说明。

计划至少包含：

```text
目标
假设
约束
步骤
每步输入
每步输出
所需工具
风险点
审批点
失败处理
停止条件
成功标准
```

支持规划模式：

```text
Direct Plan
简单问答。

Linear Plan
顺序任务。

DAG Plan
可并行任务。

Iterative Plan
探索型任务，例如排查、研究、调试。
```

要求：

1. 计划执行前必须通过 Policy Engine 检查。
2. 高风险步骤必须插入审批点。
3. 执行中发现信息不足时支持重规划。
4. 用户修改目标时支持计划更新。
5. 每一步完成后都要判断是否满足完成条件。

**十、记忆管理要求**

记忆系统必须分层设计。

至少包括：

```text
Working Memory
当前步骤临时信息。

Session Memory
当前会话上下文。

Task Memory
任务执行记录。

Long-term Memory
用户偏好、稳定知识、长期规则。

Knowledge Memory
用户导入的文档、知识库、项目资料。
```

长期记忆要求：

1. 用户可查看。
2. 用户可编辑。
3. 用户可删除。
4. 默认不保存敏感信息。
5. 写入长期记忆前要经过 Memory Policy。
6. 重要记忆写入要向用户解释原因。
7. 支持记忆过期和冲突处理。
8. 支持一键清空记忆。

**十二、安全与权限要求**

安全必须默认内建，而不是后期补。

必须支持：

1. 最小权限。
2. 按目录授权。
3. 按工具授权。
4. 按任务临时授权。
5. 高风险操作二次确认。
6. 敏感信息检测。
7. API Key 加密存储。
8. 本地数据不默认上传。
9. 用户可关闭网络访问。
10. 插件权限隔离。
11. 操作前预览影响。
12. 操作后记录审计。

高风险操作包括：

```text
删除文件
覆盖文件
批量移动文件
执行命令
安装软件
访问浏览器 Cookie
发送邮件
调用外部 API
修改系统设置
读取敏感目录
上传本地文件
```

**十三、插件系统要求**

框架应支持插件，但插件不能绕过 Harness。

插件类型：

```text
工具插件
模型插件
知识库插件
UI 插件
工作流模板插件
系统集成插件
```

插件要求：

1. 插件必须声明权限。
2. 插件必须经过用户启用。
3. 插件工具必须接入 Tool Harness。
4. 插件不能直接访问敏感资源。
5. 插件可安装、禁用、卸载。
6. 插件异常不能导致主程序崩溃。
7. 插件市场可以后续扩展，但核心框架不依赖插件市场。
